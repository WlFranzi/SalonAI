Day 24Iterate on Your Skill Until It Works

Yesterday you built a skill that works, but it might not be as good as you want it to be yet. It usually takes a few rounds of iteration to get skills to a place where they’re genuinely useful. That’s part of the process, not a problem — so today we’re going to iterate and make it better.

1.

Open the Claude desktop app and navigate to Claude Code. Select the context directory folder in the folder switcher.

Paste this prompt

Run the skill we built. After you produce the output, grade yourself: - What did you get right? - What's off? - What would you change about the skill to do better next time? Then make those changes.

2.

Add your own feedback on top of Claude’s self-assessment. Have it update the skill again.

The first version of any skill is a draft. The third or fourth version is the one that actually works.

You're done for the day. Tomorrow: making the system self-improving.

Key Takeaway

The first version of any skill is a draft. The third or fourth version is the one that actually works.

Day 25Build a Feedback Tool

You probably give feedback on the same types of things over and over - documents, proposals, designs, code reviews, student work. Today you're going to build a skill that drafts feedback in your voice.

1.

Open the Claude desktop app and navigate to Claude Code. Select the context directory folder in the folder switcher.

2.

Dictate or type something like: “I give feedback on [thing] pretty often. Usually the things I find myself saying over and over are [describe the patterns]. Can you build a skill where I input the thing I’m reviewing, and you write a first draft of the feedback you think I’d give, as close to my voice as you can?”

Not sure what to build this for? Paste this prompt: “Read my sent emails from the past week and suggest 3 things I give feedback on regularly that you could help me draft.”

3.

Review what Claude suggests. Give it any additional context about your feedback style.

4.

Tell Claude to build the skill.

5.

Now try it on something real. Upload, screenshot, or paste in a real example of the thing you give feedback on. Then say something like: “Using the skill we just built, write feedback on this.”

6.

Take the feedback Claude wrote, rewrite it so it’s closer to what you’d actually say, and paste your version back into the chat. Then say: “I rewrote the feedback you gave me. How would you change the skill to get closer to this version?” Review what it suggests, then tell it to go ahead and update the skill.

Even if this skill only gets you 80% of the way and you still make edits before sending, that’s significant time saved — you’re never starting from a blank page. And staying in the loop is often a feature, not a bug. It keeps the quality high and ensures the people receiving your feedback know there’s a real person behind it.

You're done for the day. Tomorrow: building a utility tool.

Key Takeaway

If you find yourself giving the same type of feedback repeatedly, teach Claude your patterns. It drafts, you refine.

Day 27Make the System Self-Improving

Right now, the system only gets better when you actively tell it to. Today you make improvement automatic.

1.

Open the Claude desktop app and navigate to Claude Code. Select the context directory folder in the folder switcher.

Paste this prompt

Add the following to my CLAUDE.md: ## Weekly improvement When I say "wrap up my day" on a Friday, Saturday, or Sunday, ask me if I want to do a weekly reflection. If I say yes, review my daily notes from the past week, my skills, my preferences, and my reminders. Then recommend one specific improvement to the system - a skill that should be tweaked, a preference that's changed, a workflow that's not working, a new tool to add to my toolkit, or something else worth building. Keep it to one suggestion. Explain why, and wait for me to say yes before making any changes.

2.

Now let's try it. Paste the below prompt.

Paste this prompt

Read my daily notes from the past few weeks, my skills, and my preferences. What's one thing you'd change about the system?

3.

Read the suggestion. If it's good, tell Claude to do it. If not, tell it why and ask for a different one.

The system now has a built-in improvement loop. Every week it looks at how you actually worked and suggests one change. You decide whether to make it.

You're done for the day. Tomorrow: teaching a friend.

Key Takeaway

A system that suggests its own improvements stays useful instead of going stale.

Day 29Prep Your Demo

On Day 0 you committed to a demo. Today you prep what you're going to show. First, let's make sure you can demo without sharing any private information.

Part 1: Set up recording mode

When you demo, you don't want real names, emails, or private details showing up on screen. This skill automatically anonymizes everything while you're presenting.

1.

Open the Claude desktop app and navigate to Claude Code. Select the context directory folder in the folder switcher.

Paste this prompt

Create two command files for me: 1. A file at ~/.claude/commands/recording-on.md that puts you in recording mode. When this is active, anonymize ALL identifying information in your outputs - every person's name (except mine), companies, email subjects, specific dollar amounts, and dates. Keep the structure and workflow identical, just anonymize the data. Be consistent - same real person gets the same fake name. Track the mapping internally so you can reverse it later. 2. A file at ~/.claude/commands/recording-off.md that stops recording mode. When I run this, go back to real names, identify every file that was modified during recording, and restore the real data. Show me a summary of what you restored.

2.

Type /recording-on and hit Enter.

3.

Ask Claude to show your reminders or daily note. Make sure real names are hidden.

4.

Type /recording-off to restore everything.

Part 2: Decide what to demo

Paste this prompt

Read my entire context directory - my skills, daily notes, CLAUDE.md, everything. I committed to demoing something I built. Suggest what you think I should demo and why.

5.

Review the suggestion, decide what you want to demo, and tell Claude your choice.

6.

Paste the prompt below and hit Enter.

Paste this prompt

Draft a short outline for my talk track.

7.

Review the outline.

8.

Paste the prompt below and hit Enter.

Paste this prompt

What are the attributes of an A+ demo? How would you change this talk track to meet that quality bar?

9.

Take the updated outline and make it your own. Edit it until it feels like you.

You're done for the day. Tomorrow: crossing the finish line.

Key Takeaway

Day 30Cross the Finish Line

You started this program with a browser tab and a single prompt. You're ending it with a system that plans your day, tracks your work, writes your weekly update, and tells you things about your own work patterns that you didn't notice. But this doesn't have to be the end.

1.

Open the Claude desktop app and navigate to Claude Code. Select the context directory folder in the folder switcher.

Paste this prompt

Look at the below examples of jobs that are pre-AI and post-AI. Using these examples as inspiration and what you know about my work from my daily notes, give me two lists: 1. **Before AI** - How I do my job today (be specific and honest based on what you've seen in my notes) 2. **After AI** - What my job could look like with AI fully integrated and extremely leveled up Then give me a month-long roadmap for how I get from the Before to the After over the next 30 days. Here are the examples: **Sales** - Before (~8 hrs/week): Export last month's closed-lost deals from Salesforce, read every deal note, manually notice patterns, spend hours pulling comp data, update the battlecard and hope the reps read it. After (~30 min): Your system already tagged every closed-lost deal by reason (pricing, missing feature, competitor, timing) with revenue impact attached. It caught three deals worth $420K that cited the same missing integration and drafted a product ticket. It traced a 15% drop in mid-market win rate to a competitor pricing change 3 weeks ago and updated the battlecard with counter-positioning. **Marketing** - Before (~half a day): Pull up the competitor's site, scroll their LinkedIn, check their ads library, screenshot everything, paste it into messaging docs, get your team in a room to brainstorm, brief a designer on landing page variants and wait two days, go back to the CEO with "we're working on it." After (~15 min): Your system flagged the overlap two days ago. It proposed three differentiation angles with a landing page mockup you can preview right now. It ran a conversion estimate on each option based on the last 90 days of campaign performance. You reply to the CEO with the analysis, your decision, and the reasoning behind it - before lunch. **Product** - Before (~2 weeks): Dig through Salesforce churn notes, Slack the account manager for the real story, search across support tickets and sales calls, pull a list of accounts that mentioned it, build a rough model in a spreadsheet, write a business case and circulate it, present to leadership two weeks later. After (~2 days): Your system tracks every feature request, support ticket, and sales call mentioning this capability - 34 accounts have asked, 3 have churned citing it, requests spiked 60% last quarter. It pulled the actual quotes from churned customer calls. You walk into the review with a working demo, the revenue data, the customer data, the trade-off analysis, and a clear recommendation. **Finance** - Before (~a week): Google for TAM estimates, pull numbers from three analyst reports that don't agree, build a model with one set of assumptions, email the draft to four stakeholders, get conflicting feedback, revise the model, build a 15-slide deck. After (~an hour): Your system has been tracking potential market opportunities for months. You kick off a deeper analysis - four entry scenarios, each with different assumptions on pricing, ramp, and churn. Each scenario shows exactly where it breaks. You walk into the board meeting with an interactive deck where the CFO can adjust assumptions live.

2.

Review the two lists and the roadmap. Adjust anything that doesn't feel right.

Then, when you're ready to keep going:

Paste this prompt

I just completed Hilary Gridley's Couch to 5K for AI from Lenny Rachitsky's newsletter. Go look at the website (couchto5k.ai) to get a sense of the types of daily actions I took. Now make the next 7 of those modeled after the actions in that program, personalized for me and my work.

3.

Ask Claude to add them to your calendar. Repeat for as long as you want.

Everything you built keeps getting better the more you use it.
